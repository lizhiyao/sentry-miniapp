# 一个中国小程序监控 SDK，为什么会出现在 Sentry 官方文档里

![sentry-miniapp 出现在 Sentry 官方 Community-Supported SDKs 列表（来自 docs.sentry.io/platforms/）](https://cdn.jsdelivr.net/gh/lizhiyao/sentry-miniapp@master/docs/articles/sentry-listing-screenshot.png)

前两天，我提给 Sentry 官方文档仓库的一个 PR 被合并了。

PR 很小，内容也很直接：把 `Mini Programs` 增加到 Sentry 官方文档的 Community-Supported SDKs 列表里，并把 `sentry-miniapp` 放进去。

对应的 PR 在这里：

> **PR #17152 — feat: Add Mini Programs to community-supported SDKs**
>
> <https://github.com/getsentry/sentry-docs/pull/17152>

合并之后，`sentry-miniapp` 正式出现在了 Sentry 官方文档里。

如果只把它看成一条“项目被收录”的消息，那这件事其实没什么可写的。  
但如果把它放回中国小程序生态里看，我觉得它值得单独记录一下。

## 一、为什么这件事值得记录

这些年，微信、支付宝、字节、QQ、百度、钉钉、快手等小程序平台，已经形成了一个非常大的应用生态。

它们有真实的业务场景，也有非常复杂的线上问题：

- 某个版本只在特定基础库崩
- 某个接口在弱网环境下偶发超时
- 某些错误只能在线上真机出现，开发者工具里难以复现
- 多端项目出了问题，但团队又很难在一套统一体系里排查

问题一直都在，但在国际主流监控平台的官方文档里，小程序长期没有一个明确的位置。

Web 有 Web 的接入文档，Node 有 Node 的接入文档，iOS、Android、React Native 也都有对应平台页，但轮到小程序时，开发者往往只能自己搜索、自己判断，或者默认把它当成一个“不被主流生态正式覆盖”的边缘场景。

所以这次被收录，对我来说最重要的并不是“项目多了一个背书”，而是：

**中国小程序监控，第一次在国际主流监控文档体系里有了一个清晰可见的入口。**

## 二、它的意义，不只是“官方文档里多了一行字”

`Community-Supported SDKs` 当然不等于官方自研 SDK，也不等于官方产品级承诺。

但它仍然意味着几件很现实的事。

### 1. 小程序不再完全处于文档盲区

过去你和团队讨论“小程序能不能接 Sentry”时，很多时候只能引用仓库、npm 页面、Issue，或者自己写一篇说明文档。

现在至少多了一个可以被正式检索到的官方文档入口。

这件事看起来很小，但对技术选型、方案解释、团队沟通都很重要。

### 2. `sentry-miniapp` 不再只靠 GitHub 自传播

开源项目的一个现实问题是：即使你已经做了很多年、功能也足够完整，如果没有稳定的外部入口，它依然很容易停留在“知道的人知道，不知道的人永远不知道”的状态。

被官方文档收录之后，`sentry-miniapp` 至少进入了一个更容易被发现的位置。

对于正在评估小程序监控方案的团队来说，这会带来一层额外的可信度。

### 3. 它让后续协作的门打开了一点

这次收录之后，很多后续动作都会比以前顺一些。

比如：

- 继续给 Sentry 提和小程序相关的文档或工具链 PR
- 和 Taro、uni-app 这类跨端框架讨论适配经验
- 在国内社区持续把“小程序如何进入统一监控体系”这件事讲清楚

它不是结果，更像一个起点。

## 三、这件事背后，不是一条 PR，而是很多年的持续迭代

如果 `sentry-miniapp` 只是一个刚搭起来的 Demo，这次收录其实也不会有什么说服力。

之所以敢去推动这件事，是因为在这之前，项目本身已经做了比较长时间的积累。

目前它大致具备这些基础能力：

- 基于 `@sentry/core` v10 构建，尽量沿用 Sentry 的统一 API 设计
- 覆盖 7 个小程序平台：微信、支付宝、字节、QQ、百度、钉钉、快手
- 兼容 Taro、uni-app 等跨端框架
- 支持异常自动捕获、性能监控、离线缓存、分布式追踪
- 处理了小程序 Source Map 路径归一化等落地问题
- 有相对完整的测试覆盖和持续维护

换句话说，这不是“先有收录，再补项目”，而是“项目已经做到了一个阶段，才有资格去争取被看见”。

## 四、对真实团队来说，这件事到底有什么用

如果你本来就不做小程序，那这件事确实和你关系不大。

但如果你刚好处在下面几类场景里，它会更有现实意义：

- 你在维护多端小程序，希望别再分散排查
- 你的团队已经在用 Sentry，希望把小程序也纳入同一套治理体系
- 你在做技术选型，需要一个更可信、可追溯的外部入口
- 你想把 Web、App、服务端和小程序放进同一个观测面板

它不能替你完成接入，也不能替你解决所有兼容性问题。

但它至少说明了一件事：

**小程序监控不再只是一个“只能自己摸索”的灰色地带，它开始进入主流工程体系的可见范围。**

## 五、我还在继续推进什么

这次收录之外，我也还在继续推进一些配套方向，比如和 AI 工具链、小程序集成指引、Source Map 落地体验相关的内容。

这些事都不一定会立刻见效，但它们会决定一件更长期的事：

小程序监控，能不能从“少数团队会折腾的方案”，慢慢变成“更多团队可理解、可接入、可维护的基础设施”。

## 六、最后想说一句

做开源久了会发现，最难的通常不是把代码写出来，而是让真正需要它的人看见它、理解它、敢于把它用进生产环境。

所以这次被 Sentry 官方文档收录，对我来说并不是一个“项目终于成了”的时刻，更像是“这条路开始变得更容易被别人找到”。

如果你正在做多端小程序、刚好也在用 Sentry，或者正在思考小程序监控怎么接进现有体系，欢迎来仓库交流。
